# Compliance & Target-Architecture Check

**Language:** общайся на `{communication_language}`, отчёт пиши на `{document_output_language}`.

Явная проверка инициативы/решения на соответствие регуляторике, утверждённой целевой архитектуре и принципам банка. Используется отдельно от adversarial review, когда нужен **структурированный аудиторский отчёт**, а не критический разбор.

## What Success Looks Like

К концу проверки пользователь получает **структурированный чек-лист**, в котором по каждому пункту:

- **Статус:** `compliant` / `gap` / `risk` / `n/a` / `needs-clarification`.
- **Источник:** конкретный пункт регламента / принципа / раздела target arch.
- **Доказательство:** где в решении это видно (или не видно).
- **Действие при `gap`/`risk`:** что нужно исправить и насколько срочно.

Оси проверки:

1. **Регуляторика** — 152-ФЗ (персональные данные), 115-ФЗ (ПОД/ФТ), PCI DSS (карточные данные), GDPR (если EU-периметр), внутренние политики безопасности банка.
2. **Утверждённая целевая архитектура** — соответствие распределению capabilities, owner'ам систем, паттернам интеграции из `{agent.target_architecture_path}`.
3. **Принципы банка** — `{agent.architecture_principles_path}` пункт за пунктом.
4. **Стандарты данных и интеграции** — каноническая модель, паттерны event-streaming, API contracts.

## Approach

Загрузи `{agent.compliance_corpus_path}` — может быть директорией; читай только релевантные документы (определяется доменом инициативы — карты → PCI DSS, физлица → 152-ФЗ, payments → 115-ФЗ).

Иди **по чек-листу**, не по решению. Это аудиторский режим: каждый требуемый контроль — отдельная строка отчёта.

Если требование **неприменимо** (например PCI DSS для инициативы без карточных данных) — явно пометь `n/a` с объяснением. Не игнорируй молча.

Если данных в исходнике не хватает для вынесения статуса — пометь `needs-clarification` и сформулируй точный вопрос автору. Не угадывай.

**Тон:** клинический. Здесь Archon не консультант и не критик — он аудитор. Никаких опций, никаких trade-off'ов. Только факт соответствия/несоответствия.

## Anti-Patterns

- **«В целом всё нормально».** Compliance-проверка не бывает «в целом». Каждое требование — отдельная строка.
- **Молчание про n/a.** Если PCI DSS неприменим — это всё равно строка отчёта со статусом `n/a` и объяснением.
- **Размытость источника.** «Нарушает GDPR» — не находка. «Нарушает GDPR Art. 32 (security of processing) — данные клиентов передаются между сервисами без шифрования in-transit» — находка.
- **Смешивание с adversarial review.** Архитектурные риски без явной привязки к регуляторике/target arch/принципу — не сюда. Это в `adversarial-review`.

## Completion Criteria

Проверка завершена, когда:

- Применимый периметр (домен данных, юрисдикции, регламенты) явно определён.
- Каждое применимое требование имеет статус (`compliant` / `gap` / `risk` / `n/a` / `needs-clarification`), источник и доказательство.
- Все `gap`/`risk` имеют конкретное действие и срочность.
- Вердикт сформулирован одним предложением: `can-proceed` / `conditional` / `blocked`.

Если выявлены архитектурные опции — proceed to `architectural-consultation`. Если только что-то одно жёстко нарушено и нужно показать почему — proceed to `adversarial-review`.

## Headless: `--headless:compliance-audit`

Запуск без аргументов или с указанием конкретного артефакта/директории. Прочитай landscape registry + compliance corpus, выяви gaps на уровне всего банка (не одной инициативы) и оформи отчёт в `{agent.assessment_output_path}/compliance-audit-{YYYYMMDD}.md`. Сосредоточься на high-severity gaps — детальный отчёт получится слишком большим.
